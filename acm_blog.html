#include <stdio.h>

首先出现的代码，可以注意到这里的代码，和之后代码风格完全不同，这行代码，首先它是以＃开始，中间使用<>，代码末尾也没有；结束。 于是我上网百度了一下这段代码，以下是百度百科给出的回答

＃include是在程序编译之前要处理的内容，称为编译预处理命令。编译预处理命令还有很多，它们都以“#”开头，并且不用分号结尾，所以不是c语言的程序语句。

随后我上网继续查找这个，这行代码中，include，以及stdio，甚至是"<>"这个符号的意思。 include 为文件包含命令，其意义是把尖括号或者<>内指定的文件包含到本程序中，成为本程序的一部分。被包含的文件由编译器系统提供的，其扩展名为.h 而stdio为standard input output的缩写，意为“标准输入输出” 也就是说，在c语言程序运行开始前，也就是在

int main()
      {
      return 0;
      }

这段代码运行前，c语言有一个预处理的过程，这个过程中，它会引入，你需要c语言程序执行的一些函数的文件，就比如，printf()这个函数，就是系统送你的，你只需要引入就行了，而不用自己大费周折，从头自己先def定义一个printf()的函数。 之后我们碰见的就是这段代码

int main()
      {

      return 0;
      }

这里，出现了不少东西，"int" "main" "return" "0"，以及“{};”这三个符号。 说实话一开始上来碰见这些的东西就有点蒙，听网上讲c语言的课程好像在这里通常都是让我们原封不动地抄下来，中间再加个printf("hello,world")；就结束了，没有任何的解释。 但是，单纯的抄下来，对于我，多年的理科经验告诉我。一个陌生的，不能了解透彻的，单纯只是记住的知识点，日后总是会让我吃不少苦。 于是，我又去查了一下有关这方面的知识。 根据我在五花八门的main()这篇文章中的了解，我在这里简单梳理一下。

首先，我们在各式各样的有关c语言的书上可以看见各式各样的有关于这个主函数的写法。那么有关于c语言最早的写法是什么样子的呢？？

最初main()函数的写法非常简洁，那个时候的C程序员哪怕一个字符似乎都不肯多写。不知道是因为当时键盘质量不好还是因为编辑器太糟糕的缘故，那个时代的C程序员似乎惊人地一致崇尚“简约”——甚至可以说是“至简”。

main()
      {
      printf("hello,world\n");
      }

从上面可以看出，最早的c语言程序的写法，没有返回值类型的定义“int”，没有return 0；甚至连＃include都没有。

在《The C Programming Language》的第一版中确实没有。那个时代的C语言，返回值类型为int的函数不用声明。不过在该书的第二版（1988）中这个程序被改成了

#include <stdio.h>
      main(){
      printf("hello,world\n");
      }

这里注意一下，出现＃include的写法是在1988年，也就是最早的c标准，ANSI C（也叫c89）出现的前一年，作为行业入门的教科书，里面的代码一定程度上逐渐关注代码写法的规范性，虽然我们熟悉的int main＋return 0还没有出现，但是＃include已经登场。 但是大家不要心急，int main很快就进入c语言的编写历史中。在＃include出现的不到一年后，在随后的ANSI C的标准中，返回值类型为int的函数不用声明当时编写者的这一习惯已经开始被c语言标准制定者所不倡导

返回值类型为int的函数不用声明的规则没有改变。改变了的是观念，人们已经不再倾向于代码的“至简”，而开始倾向于在代码中交代清楚每一个标识符的来龙去脉。从C89开始倡导在函数调用之前一定要有函数声明，但并没有强求，而在C99这已经是强制性的要求了。由于《The C Programming Language》第二版正值ANSI C标准颁布（1989）前夕出版，所以这种变化也应该视为ANSI C标准的倾向性以及K&R对新标准的认同。尽管这个例子没有完全反映出来这种认同。

这里出现了一个很奇怪的现象，就是为什么《The C Programming Language》第二版书中范例代码却不去按照ANSI C提倡的代码规范给出，一种“正确”的写法呢？

因为这个main()的定义并没有按照函数原型(Function prototype)的方式来写，C90中规定不带参数的main()函数应该这样写：int main(void) { /. . ./}但同时规定那个int可以省略。C90把()内不写任何内容视为过时的写法，尽管C90无奈地容忍了它(The use of function declarators with empty parentheses (not prototype-format parameter type declarators) is an obsolescent feature.)。

所以，我们简单的理一下逻辑，其实在C90时代，也就是1990年美国国家标准协会采纳为ISO/IEC 9899:1990的标准真正规范的写法应该是

#include<stdio.h>
      int main(void)
      {printf("hello,world\n")}

所以看到这里，我们已经发现了，我们的int main以及＃include都已经出炉，就剩下return 0还没出现。其实在这其中，我们也可以从中发现，其实我们写的代码中，有些时候的报错，其实是每一代语言标准制定者希望行业按照某种规范来书写代码，所以不让我们的代码通过。 对于上面，为什么C90会容忍不规范的写法，原因也很简单，积重难返。 对于return 0的出现其实在《The C Programming Language》第二版中就已经出现了，但是在这版书中，关于return的写法还出现了如下的例子

#include <stdio.h>
      main(int argc,char *argv[])
      {int found = 0 ;
      /*……计算found的值 */
      return found;
      }

在那个C99和C11还未出现的年代，这种诡异的写法是一种被允许的，它也是一种尝试。但是在如今，你如果写入你的c语言程序里面，相信你还没来得及编译的时候，就被老师发现，当作反面例子的典型，用来教育一届又一届的学弟学妹们。

int argc,char *argv[]

这两个数据，是这里出现的新玩意，简单解释一下，就是，当你在运行的程序的时候，你可以在运行的同时，就加上参数，就比如

 ~/test.EXE   a   b  c  d  

当你在你的电脑运行这样一个命令的时候

argc记录了用户在运行程序的命令行中输入的参数的个数。  arg[]指向的数组中至少有一个字符指针，即arg[0].他通常指向程序中的可执行文件的文件名。在有些版本的编译器中还包括程序文件所在的路径。

然后你电脑里面内部就是这样的

argc   =   4         
      argv[0]   =   "a"     
      argv[1]   =   "b"     
      argv[2]   =   "c"     
      argv[3]   =   "d"

让我们再回到，“return 0；”。 关于return 0，你肯定要问了，那么远古c语言程序不写return 0的时候，计算机在干嘛？？难道还自动帮程序员写这条么？？

按照C90标准，会返回一个不确定的int类型的值，如果确实不关心这个返回值是多少，不写确实可以。但C99却要求编译器在编译的时候帮忙给补上这个“return 0;”，C99在必须写int这个问题上没有迁就懒人，但在这里却对偷懒的做法给予了迁就。

现在我们重新来理整个程序的思路 首先，当我们使用我们的C语言程序的时候，用＃include预处理，然后开始调用主函数，最后主函数返回给操作系统一个值以表明程序是在何种状态下结束的，然后整个程序结束。中间其实存在着大量的底层的原理，例如编译器，操作系统在一个完整的c语言程序中是如何参与，我不得而知。这个问题可能需要留到我大二的时候，学操作系统这门课的时候再来解答了。 当然，最后的最后，通过一系列了解，我想C语言标准制定者最喜欢的写法，应该如下：

      /*  */

      }
